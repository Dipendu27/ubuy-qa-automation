# Release Notes — v2.0.2 "Portability Patch" (2026-07-10)

## Goal

v2.0.1 made green runs trustworthy. v2.0.2 makes them **reproducible anywhere** — Docker, Linux CI, and staging environments — closing audit findings H5, H6, M2, and M3.

## What changed

### 1. Docker can now actually run the suite (H5)

**Before:** `playwright.config.ts` hardcoded `headless: false`; the `HEADLESS` env var existed in `src/config/env.ts` but was never wired in (dead code). The container `CMD` launched headed Chromium with no display server → guaranteed crash on `docker run`. CI only survived because the workflow wrapped commands in `xvfb-run` itself.

**After:**
- `headless: env.headless` in `playwright.config.ts` — headed stays the default (§5.1 real-browser fingerprint for Cloudflare); `HEADLESS=true` opts into headless.
- Docker `CMD` wraps the suite in `xvfb-run --auto-servernum --server-args="-screen 0 1920x1080x24"` (xvfb ships in the Playwright base image).
- `baseURL` also now flows through the centralized `env` reader — one source of truth, no duplicated `process.env` reads.

### 2. Visual baselines are honestly platform-gated (H6)

**Before:** only `mobile-header-baseline-chromium-desktop-win32.png` was committed. Playwright suffixes snapshots per platform, so Linux/macOS runs either failed with "missing baseline" or silently wrote new ones — inside a test that also carried functional assertions.

**After:**
- The snapshot comparison is a **dedicated test** (`mobile header matches visual snapshot baseline`). If no baseline exists for the current platform it skips with an instructive reason — the functional mobile-header assertions are no longer hostage to platform drift.
- New script: `npm run snapshots:update` regenerates baselines for the responsive spec.
- Nightly workflow gains a manual `update_snapshots=true` dispatch input that generates **Linux** baselines under xvfb and uploads them as an artifact — download, commit, and the skip disappears on Linux.
- Suite count: 62 → 63 tests; traceability matrix updated.

### 3. Product discovery respects BASE_URL (M2)

`findCurrentOosProduct` no longer hardcodes `https://www.ubuy.co.in` — it builds the search URL from `env.baseUrl`, so a staging run never silently leaks to production.

### 4. OOS discovery page gets WAF/cookie protection (M3)

The `beforeAll` hook in `out-of-stock.spec.ts` previously used a raw `browser.newPage()` that bypassed the auto-fixture. It now injects the same cookie-banner suppression and runs `checkCloudflareWaf` after discovery, so a Cloudflare challenge yields an honest `environment-blocked-by-waf` skip instead of the misleading "no OOS product found".
